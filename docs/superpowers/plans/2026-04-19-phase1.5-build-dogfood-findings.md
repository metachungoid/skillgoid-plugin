## Findings

### F1. Analogue clones contaminate lint scope (synthesis-stage bug)

`scripts/synthesize/ground.py` clones analogues into `.skillgoid/synthesis/analogues/<slug>/` — *inside the project tree*. When the synthesized criteria say `ruff check .`, ruff walks the whole tree and lints the clone. Today the only reason this doesn't also produce spurious *failures* is that the analogue repo happened to be clean.

**Severity:** high. The project's own lint posture is hidden behind the analogue's posture.

**Possible fixes:**
- Clone analogues to `~/.cache/skillgoid/analogues/<slug>/` (user-global), not inside the project.
- OR: the synthesized criteria must always exclude `.skillgoid/` (append `--exclude` arg, or emit a `[tool.ruff].extend-exclude` observation).
- OR: the adapter auto-excludes `.skillgoid/` for path-walking tools.

### F2. `type: coverage` — adapter and synthesizer disagree on the contract

The synthesizer treats `type: coverage` as "invoke the `coverage` CLI with these args" (e.g., `coverage run -m pytest`, `coverage report --fail-under=100`). The adapter (`_gate_coverage` in `scripts/measure_python.py`) treats `type: coverage` as "run pytest-cov internally and compare against `gate.min_percent` (default 80)". The `args` field is **completely ignored**.

Concrete consequences on this run:
- Two "different" gates (`coverage-run` args `['run', '-m', 'pytest']` vs `coverage-report` args `['report', '--fail-under=100', ...]`) produced **identical** output (`coverage: 93.8%`) because the adapter runs the same internal flow for both.
- `--fail-under=100` was silently dropped. User-intended 100% became 80%.
- Error messages reference "pytest-cov" while the user's gate said "coverage" — confusing.

**Severity:** critical. This is the primary blocker on synthesized gates being usable for real builds.

**Possible fixes:**
- **Simpler:** teach the synthesis subagent prompt that `type: coverage` gates are `min_percent`-shaped (no args), and that literal-CLI coverage usage should be expressed as `type: run-command`. Requires schema-doc change + prompt update.
- **More faithful:** accept the literal CLI shape in the adapter — if args starts with `['run', ...]` or `['report', ...]`, run the `coverage` CLI directly and parse its output. Preserves what the subagent wrote.
- Either way: the synthesis grounding step should never emit `coverage-run` + `coverage-report` as two gates, since they're typically one build-loop semantic (run-and-threshold).

### F3. Duplicate gates from different analogue refs (grounding dedup gap)

The synthesis emitted both `coverage-run` (cited from `scripts/test`) and `coverage-report` (cited from `scripts/coverage`). Observationally valid — the analogue runs these in two steps. But in a user's build loop they collapse to one gate (fail if coverage < threshold). Phase 1.5's dedup is `(command, observed_type)`-level, which catches identical commands but not semantically-equivalent two-step sequences.

**Severity:** medium. Adds noise but doesn't break anything (both gates happen to pass-or-fail together under current adapter).

**Possible fix:** detect this pair in the synthesis subagent prompt, or post-process drafts to collapse `coverage run` + `coverage report` into one `coverage` gate.

### F4. Hint text is misleading when pytest-cov args leak through

When the adapter's internal pytest-cov call fails because `--cov=.` is an unrecognized arg (no pytest-cov installed in target), the hint says *"coverage report not generated — is pytest-cov installed in the target project?"*. Accurate at one level — pytest-cov isn't installed — but the user's gate never mentioned pytest-cov. They'd look at the hint and wonder why a `coverage run -m pytest` gate is complaining about pytest-cov.

**Severity:** medium (UX; reveals F2's root cause only to readers who know the adapter internals).

**Fix:** rolls into F2.

### F5. mypy vs ruff disagree on "the project" (tool-default inconsistency — not a synthesis bug)

Baseline run: ruff scans `.skillgoid/` analogue clones; mypy doesn't (it respects package boundaries). Not a synthesis or adapter bug per se — but means the gate set isn't internally consistent about scope on a fresh project. F1's fix also resolves this.

## Implications for v0.13 scope

The evidence-first option we settled on for v0.13 now has a clear shape:

1. **F2 (gate-adapter contract) is the biggest blocker** — any v0.13 delivery of "synthesized gates that work" has to reconcile the adapter's `type: coverage` contract with what the synthesis subagent writes. Smaller scope than Phase 2's full oracle-validation idea.
2. **F1 (analogue clone location) is a cheap first patch** — one-line-ish change to `ground.py` + a migration note for existing `.skillgoid/synthesis/` directories.
3. **F3 (gate collapsing) is a synthesis-prompt improvement, not a pipeline change** — complementary to F2.
4. **Phase 2's oracle validation is not what we needed first.** The oracle would have caught F2 (gates don't produce expected signal), but F1–F4 are observable without running the user's build — they show up the first time the adapter is asked to execute the synthesized criteria.

**Recommended v0.13 scope** (to be ratified in a follow-up spec):

- **v0.13a — Adapter/synthesis contract alignment** (F2 + F4): teach the synthesis prompt the `type: coverage` contract OR teach the adapter to honor literal coverage CLI args. Pick one, call it.
- **v0.13b — Analogue clone relocation** (F1 + F5): move clones under `~/.cache/skillgoid/` so they don't pollute the build project's lint/type scope.
- **v0.13c — Gate collapsing in synthesis** (F3): post-process drafts to merge `coverage-run` + `coverage-report` into one gate.

All three are achievable in one milestone. Phase 2 (oracle validation, context7 grounding, curated templates) stays on the roadmap but goes *after* v0.13 — evidence before abstraction.
